<objective>
Build the Resume Library feature: a multi-version resume management system with server-side migration, CRUD API endpoints, and a frontend page for listing, creating, renaming, deleting, and selecting resume versions.

Purpose: Enable users to maintain multiple resume versions (e.g., tailored for different roles) and select which one to use for analysis and cover letter generation. This is the foundation for the v2.0 resume tailoring workflow.

Output: Working Resume Library page at /resume-library, CRUD API at /api/resume-library, automatic migration of existing resume.json on first launch, and updated /api/resume endpoint that reads from the selected library version.
</objective>

Add the Resume Library backend to server/index.js. This includes migration logic, helper functions, CRUD API endpoints, and updates to existing endpoints. All work is in a single file following existing patterns.

**1a. Add library helper functions (after existing readJSON/writeJSON):**

- `readLibraryIndex()` -- reads `resume_library/index.json` from DATA_DIR. Returns `{ selected_id: null, versions: [] }` if file does not exist. Uses `fs.readFileSync` + `JSON.parse` matching existing readJSON pattern.
- `writeLibraryIndex(index)` -- writes to `resume_library/index.json` in DATA_DIR. Uses `fs.writeFileSync` + `JSON.stringify(data, null, 2) + '\n'` matching existing writeJSON pattern.
- `readResumeVersion(id)` -- reads `resume_library/<id>.json` from DATA_DIR. Validates id matches `/^[a-z0-9]+$/` to prevent path traversal (per RESEARCH.md security guidance). Returns parsed JSON or null if file does not exist.
- `writeResumeVersion(id, data)` -- writes resume data to `resume_library/<id>.json`. Same validation on id.

**1b. Add migration function (after seedDemoData call):**

Function `migrateResumeLibrary()`:
- Check if `resume_library/` directory exists in DATA_DIR. If it does, return immediately (already migrated).
- Create `resume_library/` directory with `fs.mkdirSync(libraryDir, { recursive: true })`.
- Generate an ID using existing `generateId()`.
- Read existing `resume.json` using existing `readJSON('resume.json')`. If the result is an empty array `[]` (file missing) or an empty object `{}` (file exists but empty), skip migration (no data to migrate). The check should be: `if (Array.isArray(resume) && resume.length === 0) || (typeof resume === 'object' && !Array.isArray(resume) && Object.keys(resume).length === 0)`.
- Build index object: `{ selected_id: id, versions: [{ id, name: 'Default Resume', created_at: today ISO date, updated_at: today ISO date, source_id: null }] }`.
- Write index.json and `<id>.json` to the library directory.
- Log: `'Migrated resume to library structure'`.
- Leave original `resume.json` in place (copy-not-move per RESEARCH.md).

**1c. Call migration at startup (after seedDemoData):**

Add `migrateResumeLibrary()` call AFTER the existing `seedDemoData()` call on line 106. This ensures that on fresh installs, the demo data is seeded first, then migrated into the library. Migration must run before any requests are served.

**1d. Add CRUD API endpoints (after existing /api/resume PUT route):**

GET `/api/resume-library`:
- Read library index with `readLibraryIndex()`.
- Return the full index object (includes `selected_id` and `versions` array).

POST `/api/resume-library`:
- Extract `name` from req.body. Default to `'Untitled Resume'` if not provided.
- Generate ID with `generateId()`.
- Create resume data: if `req.body.resume_data` is provided, use it; otherwise use an empty resume template `{ contact: {}, summary: '', experience: [], projects: [], education: [], skills: [] }`.
- Get current index with `readLibraryIndex()`.
- Add new version entry to `versions` array: `{ id, name, created_at: today, updated_at: today, source_id: null }`.
- If this is the first version (versions was empty), set `selected_id` to the new id.
- Write version file with `writeResumeVersion(id, resumeData)`.
- Write updated index with `writeLibraryIndex(index)`.
- Return `{ ok: true, version: { id, name, ... } }`.

GET `/api/resume-library/:id`:
- Validate `:id` matches `/^[a-z0-9]+$/`. Return 400 if invalid.
- Read version with `readResumeVersion(id)`.
- Return 404 if not found.
- Return the resume data.

PUT `/api/resume-library/:id`:
- Validate `:id` matches `/^[a-z0-9]+$/`. Return 400 if invalid.
- Read current index. Find version entry by id. Return 404 if not found.
- If `req.body.name` is provided, update the version entry's `name` and `updated_at`.
- If `req.body.resume_data` is provided, write updated resume data with `writeResumeVersion(id, resume_data)` and update `updated_at`.
- Write updated index.
- Return `{ ok: true, version: updatedEntry }`.

DELETE `/api/resume-library/:id`:
- Validate `:id` matches `/^[a-z0-9]+$/`. Return 400 if invalid.
- Read current index. Find version entry index. Return 404 if not found.
- If only one version remains (`versions.length === 1`), return 400 with `{ error: 'Cannot delete the last resume version' }`.
- Remove entry from `versions` array.
- If deleted version was the `selected_id`, set `selected_id` to the first remaining version's id.
- Delete the version file with `fs.unlinkSync`.
- Write updated index.
- Return `{ ok: true }`.

PUT `/api/resume-library/:id/select`:
- Validate `:id` matches `/^[a-z0-9]+$/`. Return 400 if invalid.
- Read current index. Verify version exists in `versions` array. Return 404 if not found.
- Set `selected_id` to `:id`.
- Write updated index.
- Return `{ ok: true, selected_id: id }`.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client->API | User-supplied resume version names cross here |
| API->filesystem | Version IDs used as filenames; must prevent path traversal |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-09-01 | Tampering | /api/resume-library/:id (all routes) | high | mitigate | Validate :id matches /^[a-z0-9]+$/ regex; return 400 for invalid IDs. IDs are generated server-side by generateId() so they always conform, but validation defends against manual URL crafting. |
| T-09-02 | Tampering | POST /api/resume-library (name field) | low | accept | Name is stored as metadata only, never used as a filename. The ID is used for all file operations. Arbitrary name strings cannot cause path traversal. |
| T-09-03 | Tampering | DELETE /api/resume-library/:id | medium | mitigate | Validate ID before fs.unlinkSync. Block deletion when only one version remains (return 400 'Cannot delete the last resume version'). |
| T-09-SC | Tampering | npm/pip/cargo installs | high | accept | No new packages are installed in this phase. All functionality uses built-in Node.js modules and existing dependencies. |
</threat_model>
