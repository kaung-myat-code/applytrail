    Add `const PdfPrinter = require('pdfmake')` is NOT the correct import for the installed pdfmake version — the current published `pdfmake` package (0.3.x) exports `{ addFonts, createPdf, setUrlAccessPolicy, setLocalAccessPolicy }` directly, not a `PdfPrinter` class. At the top of server/index.js, add `const pdfmake = require('pdfmake')` and `const { buildResumePdfDefinition } = require('./lib/pdf')`, alongside the existing requires (`generateCoverLetter`, `getProvider`, etc. at the top of the file).

    Register the built-in Roboto font once at module scope (not inside the route handler, to avoid re-registering per request). IMPORTANT — the naive recipe `pdfmake.addFonts(require('pdfmake/build/fonts/Roboto'))` throws `Error: Font 'Roboto' in style 'normal' is not defined in the font section of the document definition` against the actually-installed pdfmake@0.3.11: `require('pdfmake/build/fonts/Roboto')` returns a `{ vfs, fonts }` container object, not a flat fonts descriptor, and `addFonts()` only accepts the flat descriptor. The font file bytes (`vfs`) must be separately written into pdfmake's virtual filesystem before `addFonts` is called, or font resolution fails at render time.

    Use this verified two-step registration instead: first assign `const RobotoFont = require('pdfmake/build/fonts/Roboto')`, then call `pdfmake.addFonts(RobotoFont.fonts)` (note `.fonts`, not the whole module), then iterate `Object.entries(RobotoFont.vfs)` and for each `[filename, entry]` pair call `pdfmake.virtualfs.writeFileSync(filename, Buffer.from(entry.data, entry.encoding || 'base64'))` to write the actual font file bytes into pdfmake's virtual filesystem before any `createPdf` call happens. All three statements (the require, the `addFonts(RobotoFont.fonts)` call, and the `vfs` write-loop) belong together at module scope.

    Place this near the other one-time startup calls (`seedDemoData()`, `migrateApplications()`, etc.) before `app.listen`.

    Add two new routes, placed after the existing `app.put('/api/resume-library/:id/select', ...)` handler (~line 460) and before the Draft CRUD section:

    Route 1 — `app.get('/api/resume-library/:id/export/json', (req, res) => {...})`: destructure `{ id } = req.params`, validate with `VALID_ID.test(id)` returning `res.status(400).json({ error: 'Invalid resume version ID' })` if it fails (copy exact pattern from `GET /api/resume-library/:id`), call `readResumeVersion(id)`, return `res.status(404).json({ error: 'Resume version not found' })` if null/falsy. On success, per D-09 return the raw stored JSON unmodified: `res.attachment('resume.json').json(data)` (Express 4's `res.attachment(filename)` sets the `Content-Disposition: attachment` header, then `.json(data)` sends the body — this is the idiomatic Express 4 combination, avoids manual header-then-json ordering issues).

    Route 2 — `app.get('/api/resume-library/:id/export/pdf', (req, res) => {...})`: same `VALID_ID.test(id)` guard and `readResumeVersion(id)` 404 guard as Route 1. On success, wrap the render in try/catch: call `const docDefinition = buildResumePdfDefinition(data)`, then `const pdfDoc = pdfmake.createPdf(docDefinition, { fonts: ... })` — actually the current pdfmake API takes fonts via the module-level `addFonts` call already done at startup, not per-call, so simply `const pdfDoc = pdfmake.createPdf(docDefinition)`. Await `const buffer = await pdfDoc.getBuffer()`. Set headers before sending: `res.setHeader('Content-Type', 'application/pdf')`, `res.setHeader('Content-Disposition', 'attachment; filename="resume.pdf"')`, then `res.send(buffer)`. Since `getBuffer()` is a Promise, mark the route handler `async (req, res) => {...}` and wrap the pdfmake calls in try/catch, returning `res.status(500).json({ error: 'Failed to generate PDF', details: err.message })` on failure (matches the `{ error, details }` envelope convention already used in `POST /api/resume-library`'s validation-error response).

    Filename convention: use a fixed literal filename (`resume.json` / `resume.pdf`) for both routes rather than deriving from the version's `name` field — this avoids needing to sanitize arbitrary user-entered names for `Content-Disposition` header safety (a version name could contain characters unsafe in an HTTP header), which is out of scope for this phase's discretion notes and not called out in CONTEXT.md as a requirement.

    Font resolution note: `pdfmake.createPdf(docDefinition)` in Route 2 will only succeed if both halves of the startup font registration above ran (the `addFonts(RobotoFont.fonts)` call AND the `virtualfs.writeFileSync` loop over `RobotoFont.vfs`) and `buildResumePdfDefinition` (Task 2) sets `defaultStyle.font: 'Roboto'` on the returned document definition. If either half of the registration is skipped, or `defaultStyle.font` is missing from the document definition, `createPdf(...).getBuffer()` rejects with a "Font ... is not defined" error, which the try/catch here correctly turns into the 500 response above — but the verify step below expects a 200 with real PDF bytes, so both halves of the registration and the `defaultStyle.font` key must all be present for this route to pass its automated verification.

| Boundary | Description |
|----------|--------------|
| Client -> Express API | Browser-originated HTTP requests (fetch calls from React) reach server/index.js route handlers; :id route params and req.body fields are untrusted input |
| Express -> Filesystem | Route handlers read/write server/data/resume_library/{id}.json and applications.json using values derived from request input |

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |

| T-13-01 | Tampering | GET /api/resume-library/:id/export/json, /export/pdf (:id path param) | high | mitigate | VALID_ID regex (/^[a-z0-9]+$/) rejects any id containing '/', '..', or non-alphanumeric characters before path.join() is called in readResumeVersion(); both new export routes reuse this existing guard (Task 3), preventing path traversal to arbitrary filesystem paths |
| T-13-02 | Tampering | POST /api/applications (resume_version_id field) | medium | mitigate | resume_version_id is validated with the same VALID_ID regex when present (Task 1); it is stored only as a reference string on applications.json, never used to construct a filesystem path in this plan, so a malformed value cannot cause traversal — validation still applied defensively for consistency with existing id-handling conventions |
| T-13-03 | Denial of Service | GET /api/resume-library/:id/export/pdf (pdfmake rendering) | medium | accept | pdfmake render is synchronous/in-process CPU work with no external network calls or unbounded loops; a single-user local tool has no concurrent-request threat model that would turn this into a meaningful DoS vector -- accepted per project's no-auth, single-user constraint (CLAUDE.md) |
| T-13-04 | Information Disclosure | GET /api/resume-library/:id/export/pdf error path | low | mitigate | Catch block returns { error, details: err.message } — err.message from pdfmake/JS internals could theoretically leak file paths in a stack trace if uncaught, but Task 3 explicitly catches and returns only err.message (not the full stack), matching the existing sanitizeError-adjacent convention used elsewhere in this codebase (server/lib/analysis/providers/ai.js sanitizeError) |
