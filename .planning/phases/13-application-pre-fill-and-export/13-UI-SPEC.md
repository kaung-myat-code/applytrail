| Property | Value |

## Spacing Scale

Declared values (project's existing 8px-grid tokens — reuse verbatim, do not invent new ones):

| Token | CSS Variable | Value | Usage |
|-------|-------------|-------|-------|
| xs | `--space-1` | 4px | Badge/inline padding |
| sm | `--space-2` | 8px | Compact element spacing, button vertical padding |
| — | `--space-3` | 12px | Button horizontal padding, tight gaps |
| md | `--space-4` | 16px | Default element spacing, card gaps |
| lg | `--space-5` | 24px | Section padding, card padding |
| xl | `--space-6` | 32px | Layout gaps |
| 2xl | `--space-8` | 48px | Major section breaks |
| 3xl | `--space-10` | 64px | Page-level spacing, empty states |

Note on `--space-3` (12px): this is a pre-existing project token (not on the canonical 4/8/16/24/32/48/64 set) that this phase reuses intentionally for button horizontal padding and tight gaps, consistent with existing `.btn` usage in `ResumeLibrary.module.css`. It is not a new value invented for this phase.

Exceptions: none. Modal dialog uses the same scale (`--space-5` internal padding, `--space-4` gap between fields, `--space-6` gap between dialog and backdrop edge on small viewports).

## Typography

Reuses existing tokens exactly — no new sizes introduced.

| Role | Size | Weight | Line Height | Font |
|------|------|--------|-------------|------|
| Body / form input | 0.9rem (~14.4px) | 400 | 1.5 | `--font-body` |
| Label (button, meta, badge) | 0.85rem (~13.6px) | 600 | 1.4 | `--font-body` |
| Modal title (H2) | 1.25rem (20px) | 600 | 1.2 | `--font-body` |
| Page heading (H1, unaffected by this phase) | 2.25rem (36px) | 400 | 1.2 | `--font-display` |

Note: This new modal/export UI surface is capped at exactly 2 weights — 400 (body/form input) and 600 (labels, buttons, badges, and the modal title). The modal title uses 600 at 1.25rem rather than a heavier weight, matching the same weight already used for labels/buttons in this phase; size alone (1.25rem vs. 0.85rem) carries the title's visual hierarchy. Existing codebase does not follow a strict 3-4-size/2-weight system elsewhere (body text at 15px globally, 0.8–1.1rem for meta/labels, and 700/600/400 weights across components) — that legacy variance is unaffected by this phase and out of scope here. This phase does not need to introduce new sizes: reuse `1.25rem` (matches `.name` card headers at `1.1rem`, bumped slightly for dialog title hierarchy) and `0.9rem`/`0.85rem` (matches existing `.btn`/`.renameInput` sizing in `ResumeLibrary.module.css`).

## Color

Reuses existing tokens exactly — no new colors introduced.

| Role | CSS Variable | Value | Usage |
|------|-------------|-------|-------|
| Dominant (60%) | `--color-bg` / `--color-surface` | `#faf9f7` / `#ffffff` | Page background, modal dialog surface |
| Secondary (30%) | `--color-border-light`, `--color-border` | `#f0eef5` / `#e8e5f0` | Card borders, modal backdrop overlay tint, dividers between modal sections |
| Accent (10%) | `--color-primary` | `#5b4cdb` | Reserved for: primary CTA buttons only — "Confirm & Create Application", "Export PDF" (as the higher-intent action vs. "Export JSON"), focus rings (`--shadow-focus`) on editable modal fields |
| Destructive | `--color-danger` | `#c74040` | Reserved for: Cancel-with-unsaved-edits confirmation only if implemented (not required by CONTEXT.md — Cancel simply dismisses per D-04); do NOT apply danger styling to "Cancel" button itself (Cancel is neutral, not destructive — it discards a draft form, not existing data) |

Accent reserved for: primary CTA buttons (`Confirm & Create Application`, `Save to Library` — already primary), focus-visible rings on the two editable text inputs (company, role) and the editable cover-letter textarea, and the "Selected" state affordance already established in `ResumeLibrary.module.css`. Do not use accent color on "Export PDF"/"Export JSON" buttons — those are peer secondary actions styled like existing `.btn` (bordered, neutral), not primary CTAs, to avoid diluting the 10% accent budget on a card that already has 4-5 actions.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Modal title (auto-trigger, after Save to Library) | "Create application for this resume?" |
| Modal title (manual trigger, from ResumeLibrary card) | "Create Application" |
| Modal subtext | "Review the details below, then confirm to add this to your application tracker." |
| Company field label | "Company" |
| Role field label | "Role" |
| Status field label | "Status" (select, default `drafted`) |
| Job posting excerpt label | "Job Posting" (read-only, truncated to ~280 characters with trailing "…" if longer, per existing truncation-free precedent elsewhere in the app — this is the first excerpt/truncation UI, so cap explicitly to avoid an unbounded modal) |
| Linked resume version label | "Resume Version" (read-only, shows version name, e.g. "Meridian Software - Fullstack Developer") |
| Cover letter field label | "Cover Letter (editable)" |
| Cover letter loading state | "Generating cover letter..." |
| Cover letter generation failure | "Couldn't auto-generate a cover letter. You can write one manually below, or leave it blank and add it later." (field remains editable and empty — does not block Confirm) |
| Primary CTA (confirm) | "Confirm & Create Application" |
| Primary CTA loading state | "Creating..." (button label swap, disabled during request — matches "Exporting..."/"Saving..." pattern) |
| Secondary action (cancel) | "Cancel" |
| Post-confirm success (auto-trigger path) | Toast/inline: "Application created. Redirecting to Resume Library..." then navigate to `/resume-library` |
| Post-confirm success (manual-trigger path) | Toast/inline: "Application created." then navigate to `/applications` (surfaces the user's new record directly, since they initiated from the library, not from a save flow already headed there) |
| Confirm error (application creation fails) | "Couldn't create the application. [server error message]. Try again." — modal stays open, form state preserved |
| Cancel — auto-trigger path | Dismisses modal, navigates to `/resume-library` (resume was already saved per D-04 — cancel only skips application creation) |
| Cancel — manual-trigger path | Dismisses modal, stays on `/resume-library`, no navigation |
| Export PDF button (card action) | "Export PDF" |
| Export JSON button (card action) | "Export JSON" |
| Export PDF loading state | "Exporting..." (button label swap, matches existing `Creating...`/`Saving...` in-progress pattern) |
| Export failure (either format) | Inline error near the card: "Couldn't export this resume as PDF/JSON. Try again." (reuse `.error` styling from `ResumeLibrary.module.css`) |
| Destructive confirmation | Not applicable — neither pre-fill confirm nor export are destructive actions. Existing delete confirmation (`window.confirm('Are you sure you want to delete this resume version?')`) is unchanged by this phase. |

## UI Considerations

Applicable state considerations resolved: 10 covered, 3 backstop, 0 unresolved

| Category | Element(s) | Status | Resolution / Reason |
|----------|------------|--------|---------------------|
| loading | Cover letter field (modal) | ✅ covered | Field shows "Generating cover letter..." placeholder text while `POST /api/generate-cover-letter`-equivalent call is in flight; Confirm button is not disabled during this wait (user may confirm with an empty/partial cover letter, per D-05 discretion) |
| error | Cover letter auto-generation failure | ✅ covered | Field falls back to empty + editable, inline note per Copywriting Contract row above; does not block modal Confirm |
| error | Application creation POST fails | ✅ covered | Modal stays open, inline error shown, form values preserved for retry (see Copywriting Contract "Confirm error" row) |
| error | Export PDF/JSON fetch fails | ✅ covered | Inline error near the resume card, matches existing `.error` class; export buttons return to idle label ("Export PDF"/"Export JSON") for retry |
| empty | Job posting excerpt when `posting_text` is empty/missing | 🧪 backstop | Show literal "No job posting text available." in the read-only field rather than blank space — this is an edge case not explicitly discussed in CONTEXT.md (all documented flows assume a posting exists via `draft.posting_id`); planner/executor should treat this as a defensive placeholder, not a blocking condition |
| overflow | Job posting excerpt exceeds ~280 chars | ✅ covered | Truncate with trailing "…", per Copywriting Contract row — prevents unbounded modal height |
| overflow | Resume version name is very long (long company/role concatenation) | 🧪 backstop | Apply `text-overflow: ellipsis` with `white-space: nowrap` + `overflow: hidden` on the read-only resume-version-name display, consistent with `.name` card header behavior; no explicit max-width was specified in CONTEXT.md, so this is a defensive CSS backstop, not a confirmed design requirement |
| zero-one-many | ResumeLibrary card actions row with 5 actions (Select, Rename, Delete, Edit, + new Export PDF, Export JSON, Create Application = 8 total) | 🧪 backstop | Existing `.actions` flex row (`ResumeLibrary.module.css`) uses `flex-wrap`-free horizontal layout; with 3 new buttons added, row may overflow on narrow viewports. Backstop: add `flex-wrap: wrap` to `.actions` so buttons reflow to a second line rather than causing horizontal scroll or clipping. No responsive breakpoint was specified in CONTEXT.md. |
| long-text | Company/Role inline-editable fields (modal) | ✅ covered | Standard text input, no character limit specified — reuse existing `.renameInput`-style input sizing (`min-width: 200px`), allow natural text wrapping is not applicable to single-line inputs; horizontal scroll within input is acceptable/standard browser behavior |
| loading | PDF export while pdfmake generates document server-side | ✅ covered | Button shows "Exporting..." per Copywriting Contract; button disabled during request to prevent duplicate triggers |
| populated | Confirmation modal — happy path, all fields present | ✅ covered | Company, role, job posting excerpt, resume version name, and generated cover letter all render pre-filled per D-03/D-05 |
| loading | Confirm button while `POST /api/applications` is in flight | ✅ covered | Button disables and swaps label to "Creating...", same disabled+label-swap pattern already used for "Export PDF"/"Export JSON" (Exporting...) and `PreviewTailored.jsx`'s Save button (Saving...) — prevents duplicate submission |
| partial | Draft has company/role but posting was deleted between analysis and save | 🧪 backstop | Not explicitly covered in CONTEXT.md (edge case of concurrent data deletion in a single-user local tool is low-probability); treat same as empty-posting backstop above — show "No job posting text available." rather than erroring the whole modal |

## Modal Interaction Contract

(Supplementary section — not in template, but required because this is a net-new UI pattern with no codebase precedent for the planner/executor to reference.)

| Property | Value |
|----------|-------|
| Trigger (auto) | Immediately after `POST /api/drafts/:id/save` resolves successfully in `PreviewTailored.jsx` (before/instead of the existing `navigate('/resume-library')` call) |
| Trigger (manual) | "Create Application" button on a resume version card in `ResumeLibrary.jsx` |
| Backdrop | `position: fixed`, full viewport, `background: rgba(26, 22, 37, 0.4)` (derived from `--color-text` `#1a1625` at 40% opacity — no existing overlay token, this is the first one; do not invent a new CSS variable, use this literal rgba consistent with the app's warm-dark palette) |
| Dialog surface | `--color-surface` (`#ffffff`), `border-radius: var(--radius-lg)` (14px), `box-shadow: var(--shadow-lg)`, max-width 560px, centered, `padding: var(--space-5)` |
| Dismiss affordances | Cancel button (primary dismiss path per D-04) + Escape key + backdrop click. All three trigger the same Cancel behavior (no unsaved-changes warning — CONTEXT.md does not specify one, and the modal's only persisted side effect happens on Confirm) |
| Focus management | Focus moves to the first editable field (Company input) on open; focus returns to the triggering button on close (standard dialog accessibility baseline — not explicitly requested in CONTEXT.md but required for no-regression accessibility since this is the app's first modal) |
| Scroll behavior | Page scroll locked while modal is open (`overflow: hidden` on body or equivalent) since this is a blocking confirmation step, not a non-modal panel |
| Animation | Reuse existing `fadeInUp` keyframe pattern already defined in `index.css`/`ResumeLibrary.module.css` (`--duration-normal` / `--ease-out`) for the dialog entrance; simple opacity fade for the backdrop |
| Visual focal point | The primary CTA button ("Confirm & Create Application") is the visual anchor of the dialog — bottom-right position, `--color-primary` fill, standard button elevation; the cover-letter textarea is the secondary focus (largest field, positioned above the CTA row) since it is the one field users are most likely to read and edit before confirming |
