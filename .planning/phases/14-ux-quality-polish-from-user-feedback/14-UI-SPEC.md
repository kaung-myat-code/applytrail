# Phase 14 — UX Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

This phase is **UX polish on shipped surfaces**, not new-screen design. Every decision below extends the existing token system and component patterns already established in `client/src/index.css` and confirmed by the Phase 13 UI-SPEC precedent (`.planning/phases/13-application-pre-fill-and-export/13-UI-SPEC.md`). No new colors, spacing units, fonts, or component libraries are introduced.

**In scope for this contract (UI-facing decisions from `14-CONTEXT.md`):** D-01, D-02, D-04, D-05, D-06, D-07, D-11.
**Out of scope for this contract (backend/text-generation/tooling, no visual contract needed):** D-08 (resume-library default fix), D-09/D-10 (tests), D-12 (lint), D-13 (keyword whitelist expansion — logic only). D-14 and D-15 touch rendered text (cover letter, suggestions, keyword badges) and are addressed narrowly under Copywriting Contract below where they affect what the UI displays, not how it should generate that text.

## Typography

Reuses existing tokens exactly — no new sizes introduced. New UI this phase (nav dropdowns, date-clarity labels, continue-CTAs, save/unsaved indicator, delete-confirm text, preview modal) is capped at 2 weights, matching the Phase 13 precedent.

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Body / preview content | 0.9rem (~14.4px) | 400 | 1.5 | Preview modal resume body text (reused from `PreviewTailored.jsx` rendering, wrapped in modal) |
| Label / nav / meta / CTA | 0.85rem (~13.6px) | 600 | 1.4 | Nav top-level group labels, nav dropdown items, "Continue to [next step]" CTA text, Applications date-clarity labels, Saved/Unsaved indicator text |
| Small meta | 0.8rem (12.8px) | 600 | 1.4 | Existing `.label` uppercase field-label style (`Resume.module.css`) — reused unmodified for delete-confirm inline copy |
| Section heading (unaffected) | 1.5rem (24px) | — | 1.2 | `--font-display`, unchanged by this phase |

Note: existing `Navbar.module.css` `.link` already uses weight 500 (inactive) / 600 (active). This phase keeps that exact pattern for top-level nav items and dropdown children — do not introduce a third weight.

## Color

Reuses existing tokens exactly — no new colors introduced.

| Role | CSS Variable | Value | Usage in this phase |
|------|-------------|-------|-------|
| Dominant (60%) | `--color-bg` / `--color-surface` | `#faf9f7` / `#ffffff` | Page backgrounds, nav dropdown panel surface, preview modal surface |
| Secondary (30%) | `--color-border-light` / `--color-border` | `#f0eef5` / `#e8e5f0` | Nav dropdown panel border, divider between date-clarity labels, preview modal section dividers |
| Accent (10%) | `--color-primary` | `#5b4cdb` | Reserved for: "Continue to [next step]" CTA text/icon, active nav group label + open dropdown indicator, "Confirm & Save" button in the Save Application inline confirm, "Preview" button in the resume editor, focus rings on all new interactive elements |
| Warning | `--color-warning` / `--color-warning-bg` | `#c07d2b` / `#fef9ee` | "● Unsaved changes" indicator (repurposes existing warning token — this state is informational, not an error) |
| Success | `--color-success` | `#2d9d78` | "✓ Saved" indicator (reuses existing `.savedMessage` color unchanged) |
| Destructive | `--color-danger` / `--color-danger-bg` | `#c74040` / `#fef2f2` | Delete-confirm affordance (reuses existing `.removeButton`/`.removeEntryButton` hover colors, unchanged — `window.confirm()` dialog itself is native browser chrome, not styled) |

Accent reserved for: `Continue to [next step]` CTA links/buttons only (not every link — plain navigational `Link`s like "Back to Analysis" stay `--color-muted`/text-colored per existing `.backLink`/`.backButton` styles), the active/open state of a nav group label, the "Confirm & Save" primary button in the Cover Letter save-confirmation step, and the "Preview" trigger button on the Resume editor. Do NOT apply accent to the nav dropdown's inactive child items (they use the existing `.link` muted/hover treatment, only the active child gets `--color-primary-light` background per current `.link.active`).

## Interaction Contracts

(Supplementary — required because three of this phase's fixes are net-new interaction patterns with no full precedent in the codebase.)

### Nav dropdown (D-05/D-07)

| Property | Value |
|----------|-------|
| Trigger | Click on group label (not hover — matches the click-based, keyboard-accessible pattern already used for the app's one existing overlay, the modal's Escape-to-close) |
| Panel | Absolutely positioned below the trigger, `background: var(--color-surface)`, `border: 1px solid var(--color-border-light)`, `border-radius: var(--radius-md)`, `box-shadow: var(--shadow-md)` — new but composed entirely from existing tokens |
| Open/close | One group open at a time; opening one closes any other open group; closes on: selecting a child link, Escape key, or clicking outside the panel (outside-click and Escape handlers mirror the pattern in `CreateApplicationModal.jsx`) |
| Active state | Group label styled `--color-primary` / `--color-primary-light` background when the current route matches a child (see Copywriting Contract) — the group itself is never a navigable route |
| Standalone items | "Dashboard", "New Application", "Applications" remain plain `NavLink`s exactly as today (no dropdown) |
| Mobile (≤640px) | Existing `Navbar.module.css` mobile breakpoint stacks nav items vertically (`flex-direction: column`); dropdown panels render inline/expanded (not floating) below their trigger at this breakpoint, consistent with the existing stack-not-overlay mobile behavior |

### Save Application inline confirm (D-02)

| Property | Value |
|----------|-------|
| Trigger | "Save Application" button click (only enabled once a cover letter has been generated, same gating as today) |
| Expanded state | Button is replaced in place by a confirm row: prompt text + "Confirm & Save" (accent, primary) + "Cancel" (neutral, matches `.btn` from `CreateApplicationModal.module.css`) |
| No modal | This is a deliberately lighter-weight pattern than `CreateApplicationModal` — no backdrop, no field editing, since all data (posting, cover letter) is already visible on the page. Reserve the full modal pattern for flows that need field review/editing (pre-fill, D-13's precedent) |
| On confirm success | Auto-navigate to `/applications` — no lingering inline success message (removes the current dead-end "Application saved — View in Applications" link, since D-02's entire point is to stop leaving the user uncertain) |
| On confirm failure | Row stays expanded with error text below it, "Confirm & Save" re-enabled for retry |

### Resume preview modal (D-11)

| Property | Value |
|----------|-------|
| Trigger | "Preview Resume" button in `.saveRow`, next to Save |
| Backdrop/dialog | Reuse `CreateApplicationModal.module.css` `.backdrop`/`.dialog` classes verbatim (same `rgba(26, 22, 37, 0.4)` overlay, same `--radius-lg`/`--shadow-lg` dialog, same `fadeInUp`/`fadeInOpacity` entrance animation) |
| Content | Reuse the read-only resume rendering structure already written in `PreviewTailored.jsx` (contact line, Summary, Skills badges, Experience/Projects/Education entry cards) — render the current in-editor `resumeData` (including unsaved edits), not the last-saved version, so Preview reflects exactly what the user is about to save |
| Dismiss | "Close" button, Escape key, backdrop click — page scroll locked while open (`document.body.style.overflow = 'hidden'`, same as `CreateApplicationModal`) |
| Focus | Focus moves to the "Close" button on open; returns to the "Preview Resume" trigger on close |

## UI Considerations

Applicable state considerations resolved: 9 covered, 2 backstop, 0 unresolved

| Category | Element(s) | Status | Resolution / Reason |
|----------|------------|--------|---------------------|
| empty | Resume Preview modal when a section (e.g. Projects, Education) has no entries | ✅ covered | Reuse `PreviewTailored.jsx`'s existing conditional rendering (`{section.length > 0 && ...}`) — sections with no data simply don't render, matching current preview behavior exactly |
| loading | Save Application confirm row while `POST /api/applications` is in flight | ✅ covered | "Confirm & Save" swaps to "Saving..." and disables, same disabled+label-swap convention as every other async button in the app (`Creating...`, `Exporting...`, `Saving...`) |
| error | Cover Letter save-confirmation POST fails | ✅ covered | Inline confirm row stays open, error text shown below it, "Confirm & Save" re-enabled — no full-page or modal error state |
| zero-one-many | Nav dropdown with exactly 2 children per group (Resume/Resume Library, Analysis/Cover Letter) | ✅ covered | Panel sizes to content, no scroll needed at this item count; if a future phase adds a 3rd child, the existing `--space-2`/`--space-3` padding pattern scales without layout change |
| overflow | Nav dropdown panel near right edge of narrow viewports | 🧪 backstop | No explicit right-edge collision handling specified in CONTEXT.md; apply a defensive `right: 0` anchor on the last group's panel (Tailor) so it doesn't clip off-screen, consistent with standard dropdown behavior — not a confirmed design requirement, just a sane default |
| long-text | Applications list date-clarity labels on narrow viewports | ✅ covered | Both labels already stack via existing `.meta` flex-wrap behavior in `Applications.module.css` — no change needed, just longer label text within the same container |
| populated | Resume editor with Unsaved/Saved indicator after every field-level edit (name, contact, summary, every experience/project/education field, every bullet) | ✅ covered | Single source of truth: any `handleFieldChange`/`handleContactChange`/`handleExperienceChange`/etc. call marks state dirty; only a successful `handleSave` response clears it — no per-field granularity needed |
| destructive-undo | Deleting an experience/project/education entry or bullet before the next Save | ✅ covered | `window.confirm()` per Copywriting Contract; since nothing is persisted until "Save" is clicked, a cancelled edit session (navigate away without saving) already fully reverts the deletion — no additional undo affordance needed beyond the confirm dialog |
| partial | Resume Preview opened with a field mid-edit (e.g. cursor in a textarea) | 🧪 backstop | Not explicitly covered in CONTEXT.md; treat as: Preview renders the current React state as-is (including the in-progress value of a focused field, since `onChange` already updates state on every keystroke) — no special-casing needed, this falls out naturally from reusing live `resumeData` |
| loading | Resume Preview modal open/render | ✅ covered | No network request — modal renders synchronously from already-loaded `resumeData`, no loading state needed (unlike `CreateApplicationModal`'s cover-letter fetch) |
