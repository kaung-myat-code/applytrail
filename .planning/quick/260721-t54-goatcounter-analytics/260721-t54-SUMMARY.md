**Env-gated GoatCounter pageview analytics injector — disabled by default on local dev, enabled in production via VITE_GOATCOUNTER_SITE**

## Accomplishments
- `client/src/lib/analytics.js` exports `initAnalytics()`, which reads `import.meta.env.VITE_GOATCOUNTER_SITE` and only injects the GoatCounter `count.js` script when the site code is set — local dev stays untracked by default
- Idempotency guard prevents duplicate script injection on repeat calls
- `main.jsx` calls `initAnalytics()` once at app startup, with no arguments and no router changes
- `client/.env.example` and README's Deployment env-var list both document `VITE_GOATCOUNTER_SITE`, including the build-time/Render note (Vite inlines `VITE_*` vars at build)

## Decisions Made
- Chose a runtime script-injector module (called from `main.jsx`) over an inline `<script>` in `index.html`, because `index.html` is static and cannot read `import.meta.env` — the helper module keeps the env-gate logic in one testable place (per plan's design_notes).
- Kept scope to a simple pageview counter on page load; SPA route-change counting was explicitly out of scope for this quick task.

## User Setup Required

**External service requires manual configuration on Render (or wherever this is deployed):**
- To enable analytics in production, add a `VITE_GOATCOUNTER_SITE` environment variable in the Render service settings (or `render.yaml`), set to your GoatCounter site code (create a free account at https://www.goatcounter.com if you don't have one).
- Since Vite inlines `VITE_*` vars at build time, this must be set before the build step runs, and a rebuild/redeploy is required after adding or changing it.
- No action needed for local development — leaving the var unset keeps analytics disabled.
